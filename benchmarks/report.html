Bachelor thesis · Image → Structured CAD

Two specialists beat one
generalist at floor-plan understanding.

The thesis tests a simple claim: a division of labour wins. A dedicated segmentation model (Mask2Former) owns geometry — the "where". A vision-language model (Qwen3-VL-8B) owns semantics — the "what". Together this hybrid is measured against a single monolithic frontier model (Claude Opus 4.8) that tries to do both in one pass.

The idea

Why split the problem in two?

Reading a floor plan is really two different jobs. One is perception: tracing the exact pixels of every wall, room, door and window. The other is interpretation: knowing that a region is a "bedroom" or a "sauna". These need different tools — so the pipeline uses a different model for each, and the benchmark checks whether that beats handing everything to one large model.

The hypothesis is that geometry should go to the model that actually perceives pixels, semantics to the model that understands language — and that this is both more accurate and cheaper than asking one frontier model to do everything.
Benchmark 1 + 2 · Geometry (the "where")

Does the segmentation model earn its place?

Two questions. First: does fine-tuning Mask2Former actually buy anything over an untrained model and the established CubiCasa baseline? Second: can a frontier VLM (Claude) just replace it? Both are scored by pixel IoU against RLE ground truth (1.0 = perfect overlap).

Training is justified. The untrained model scores ~0 on every class — generic COCO categories simply don't transfer to schematic line drawings. Geometry belongs to the segmentation model. The gap is large and consistent: ~0.22 IoU on rooms and ~0.50 on walls. Walls are the clearest failure — Claude emits a few coarse rectangles (~0.25) where Mask2Former traces the actual wall skeleton (~0.75). A VLM estimates polygons by reasoning over the image; it does not perceive them pixel-by-pixel.
Benchmark 3 · Out-of-distribution

What happens on a drawing style it has never seen?

The geometry models are pushed onto FloorPlanCAD — colored Chinese CAD drawings with dimensions and furniture, nothing like the training data. This dataset annotates symbols on lines, not filled room polygons, so there is no comparable room-IoU ground truth. The table reports pixel coverage % (how much each model paints), not accuracy — the real signal is which model still finds structure and which collapses.

Benchmark 4 + 5 · Semantics (the "what")

Reading room types — prompt design, then the real fight

Semantics is scored as F1 on canonical room types (bedroom, kitchen, bath…) against the model.svg labels. First we tune the prompt for the small Qwen model; then we put its best version head-to-head against the frontier model on the identical 50 plans.

4 · Prompt ablation (Qwen3-VL-8B, zero-shot)

CubiCasa5k val · n=50 · same model & GT, only the prompt changes · source: SEMANTIC_RESULTS.md
prompt level L1 → L4
point size ∝ F1 · up-and-right is better
The Finnish glossary is the single biggest lever (level 1→2: +114% F1) — room-type ID is largely an OCR + translation task. A strict JSON schema then adds precision (level 3, the winner). Notably, injecting the pipeline's own geometry into the prompt (level 4) does not help: abstract region letters don't map to visible labels. Each model wins at its own job — geometry doesn't rescue semantics.

The decisive hybrid win is geometry (§2), not semantics.

Opposite failure modes. Qwen under-counts (omits ~1.7 rooms/plan, conservative); Claude over-counts (hallucinates ~1.2 rooms/plan). This is the mechanical reason behind Claude's higher recall in §5 — it's bought with more spurious rooms.

Qwen still has room to grow. The small model keeps pace with the frontier model on the common types (living, kitchen, bedroom) and even edges ahead on entry — but it clearly trails on the rarer ones: study (0.22 vs 0.80), garage (0.33 vs 0.71), bath. These long-tail labels are exactly what a future fine-tune would target to close the gap. Neither model emits a generic "other" label, so both score near-zero on the GT catch-all — a vocabulary artefact, not a real perception failure.
Economics

Reality check. That $8,800 you'd hand the API to label 100,000 plans is almost exactly the sticker price of one NVIDIA RTX PRO 6000 Blackwell — 96 GB VRAM, ~$8,565 MSRP (2026). For what a single batch costs on the API, you could buy the card outright and run the 8B model on your own desk: at ~10 s/plan it clears ~3 million plans a year on roughly $25 of electricity per 100k. The API invoice returns with every batch; the GPU is yours to keep.
Cheaper and better. At 100k plans the hybrid saves ~$7,400. And Mask2Former's geometry — the part Claude is worst at — is the part that runs for free locally. The expensive model is paying a premium to do the job it does least well.

Conclusion

The verdict, role by role

The monolithic frontier model loses decisively on geometry and wins modestly on semantics. Because the large, consistent gap lives in geometry — and because the 8B semantic model is cheap, local, and close behind — the hybrid is the right architecture for a local, single-user application. Each component is justified on the role it owns.

RoleWinnerEvidence
Geometry (where) Mask2Former — hybrid IoU room 0.889 vs 0.672, wall 0.749 vs 0.248
Semantics (what) Claude by ~0.07 F1 0.520 vs 0.451, p=0.012 (significant); Qwen close
Training value fine-tuned » untrained, > CNN beats baseline on every class
OOD robustness fine-tuned > CNN walls 11% vs 4% coverage on unseen style
Cost hybrid ~6× cheaper $0.014 vs $0.088 per image
Division of labour wins. 4 of 5 dimensions go to the hybrid; the one it concedes (semantics) it loses by a hair while being a fraction of the cost. A frontier model is impressive but it is the wrong, expensive tool for pixel-precise geometry — exactly the job a small specialist does best.
